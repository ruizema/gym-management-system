Hypothèses

J'ai supposé que le centre sportif #gym possède un système central (le centre de données #GYM) qui enregistre des nouveaux membres et des professionnels par l'intermédiaire d'un agent. Ce centre de données permet au membre de consulter et s'inscrire et comfirmer sa présence à un service offert. Il permet aussi à un professionnel de créer une séance et consulter les inscriptions.

Tâche 1 : Glossaire

Client: Personne étant intéressé à être membre de #GYM
Professionnel: Personne offrant les services au #GYM
Numéro du professionnel: Numéro identifiant du professionnel
Numéro du membre: Numéro identifiant du membre
Centre de Données: Système central qui enregistre les données personnelles des membres, les séances de services et les inscriptions aux séances. Il valide aussi les numéros de membres et les inscriptions
Agent: Personne à la réception qui se charge de transmettre toutes les informations données par le membre ou professionnel à l'ordinateur, au logiciel
Réception: Endroit où il faut passer pour faire des inscriptions, valider son numéro d'identifiant, valider ses inscriptions et tout autre informations concernant les services offerts du #GYM, ainsi que pour accéder le #GYM
Inscriptions de membres: Champs d'informations contenant les informations personnelles des membres
Inscriptions aux séances: Champs d’informations contenant les numéros d’identification du member inscrit, celui du professionnel et les informations de la séance
Répertoire de services: Fichier contenant tous les codes de services offerts
Séances: Activité proposée ou supervisée par un professionnel à des horaires fixes
Services: Activité offert par #GYM soit les salles d’entrainement, la piscine, les salles multisports et les séances
Code du service: Numéro identifiant les services
Code de séance: Numéro identifiant les séances

Tâche 3: Cas d'utilisation

Modification des cas d'utilisation en caractères gras 

        Cas d'utilisation : Création d'un nouveau compte.
        But : Le client désire adhérer a #GYM.
        Préconditions : Le client n'est pas membre de #gym.
        Acteurs : Agent (principal), Client ou Professionnel (secondaire).
        Scénario principal :
        1. Le client ou le professionnel se présente à la recéption du centre.
        2. L'agent lui demande ses informations personnelles
            2.2. L'agent rentre les informations dans son ordinateur.
            2.3. les informations sont donc enregistrées dans le système central: Le Centre de Données #GYM.
        3. Le client paie ses frais d'adhésion.
        4. Le client reçoit son numéro de membre.
        Scénarios alternatifs :
        2.3a.1 Le logiciel refuse de créer un nouvel enregistrement, le client est déjà membre.
        3a.1 La méthode de paiement est refusée ou les fonds sont déficients.
        Postconditions : Le systeme assigne un numéro unique au membre ou au professionnel.
    
        Cas d'utilisation : Validation du numéro de membre .
        But :  Le client a l'intention d'accéder au centre sportif et ses services.
        Préconditions : Le client est déjà membre du #GYM.
        Acteurs : Agent (principal), Client (secondaire) ou Professionnel (secondaire).
        Scénario principal :
        1. Le client se rend à la réception et donne son numéro d'identifiant.
        2. L'agent entre le numéro dans le logiciel.
        3. Le logiciel confirme le numéro et affiche "Validé".
        Scénarios alternatifs :
        3a.1. Le logiciel rejete le numéro et affiche "Numéro invalide".
        3b.1. Le logiciel rejete le numéro et affiche "Membre suspendu".
        3b.2. L'agent informe le membre que son compte est suspendu à cause de frais due.
        Postconditions :Le système ouvre le passage au #GYM.
    
        Cas d'utilisation : Consultation du Répertoire de Services
        But : Pour s'inscrire à une séance, l'agent doit aider le client à consulter l'ensemble des services offerts et les séances.
        Préconditions : Le client est déjà membre de #GYM.
        Acteurs : Agent (principal), Client (secondaire).
        Scénario principal :
        1. L'agent à la réception demande au logiciel de consulter la liste des services offerts.
        2. Le système fait afficher l'ensemble des services offerts et les séances disponibles dans le Répertoire des Services.
        Scénarios alternatifs :
        N/A
        Postconditions : Le système a affiché les détails des services offerts et séances disponibles. 
        Il sera ensuite possible de sélectionner une séance pour s'y inscrire.
    
        Cas d'utilisation : Inscription à une séance
        But : Le client désire s'inscrire à une séance.
        Préconditions : Le client est déjà membre de #GYM.
        Acteurs : Agent (principal), Client (secondaire).
        Scénario principal :
        1. L'agent appelle Validation du numéro de membre.
        2. L'agent appelle Consultation de Répertoire de Services.
        3. L'agent sélectionne la séance désirée et y inscrit le membre.
        4. Lors de la confirmation de l'inscription à la séance, le logiciel enregistre les champs d'inscription dans le centre de données.
        5. Le logiciel cherche et affice le montant due pour la séance.
        6. Le membre paie pour son inscription.
        Scénarios alternatifs : 
        6a.1 Les fonds sont insufficents, le membre n'est pas inscrit.
        Postconditions : L'inscription est enregistré dans le centre de données.
    
        Cas d'utilisation : Création d'une séance
        But : Le professionnel désire fournir une nouvelle séance d'un service aux clients du #GYM.
        Préconditions : Le professionnel s'est déjà inscrit à un compte au #GYM.
        Acteurs : Agent (principal), Professionnel (secondaire).
        Scénario principal :
        1. Le professionnel se présente à la réception.
        2. L'agent demande au professionnel les informations nécessaires pour créer une nouvelle séance dans le Répertoire des Services.
        3. L'agent envoie ces informations au Centre de Données.
        4. Le Centre de Données crée les nouvelles séances dans le Répertoire des Services.
        Scénarios alternatifs :
        4a.1. Le Centre de Données rejette la requête, car certaines informations fournie sont invalides.
        Postconditions : Une nouvelle séance a été créée dans le Répertoire des Services.
    
        Cas d'utilisation : Confirmation de présence à une séance
        But : Après que le client s'est inscrit à une séance, et avant que cette dernière commence, le client doit se présenter
        à la réception pour confirmer sa présence.
        Préconditions : Le client s'est déjà inscrit à la séance. La séance n'est pas encore commencée.
        Acteurs : Agent (principal), Client (secondaire).
        Scénario principal :
        1. Le client se présente à la réception et indique à l'agent la séance à laquelle il s'est inscrit.
        2. L'agent cherche la séance dans le Répertoire de Service.
        3. Le client donne à l'agent son numéro de membre.
            3.1 L'agent envoie ce numéro au Centre de données pour vérifier si l'inscription est valide.
        4. Le système indique que l'inscription est valide.
        5. L'agent donne au client accès à la séance.
            5.1 Le système crée un enregistrement de la confirmation sur le disque, avec les informations nécessaires.
        Scénarios alternatifs :
        4a.1. Le système indique que l'inscription n'est pas valide.
        4a.2. Le client est refusé accès à la séance.
        Postconditions : Le client est donné accès à la séance pour laquelle il s'est inscrit.
    
        Cas d'utilisation : Consultation des inscriptions à une séance
        But : Le professionnel désire consulter les inscriptions à ses séances.
        Préconditions : Le professionnel a déjà ouvert une séance dans le Répertoire de Services.
        Acteurs : Agent (principal), Professionnel (secondaire).
        Scénario principal :
        1. Le professionnel donne son numéro à l'agent de la réception.
        2. Le professionnel sélectionne à l'agent la séance qu'il veut consulter.
        3. L'agent envoie un requête au système pour afficher la liste des inscriptions pour la séance sélectionnée.
        Scénarios alternatifs :
        1a.1. Le numéro de l'agent est invalide.
        2a.1. Le professionnel n'a pas encore créé de séances.
        Postconditions : Le système affiche une liste des inscriptions pour la séance sélectionnée.

Tâche 4: Risques

Du plus sévère au moins sévère:

  1. Le logiciel prend plus de place de mémoire que contient l'ordinateur.
  2. Le logiciel ne donne pas de numéro d'identifiant lorsqu'on rentre les données d'un client.
  3. Lors des additions de membres, le logiciel pourrait écraser les données des membres existants.
  4. Le Répertoire de Services créer une base de données.
  5. Les programmeurs de l'équipe n'ont pas d'expérience avec le langage de Java.

Tâche 5: Exigences non-fonctionnelles

  1. Logiciel doit être écrit en Java
  2. Le Répertoire de Services doit être un fichier
  3. Il n’y a pas de bases de données ou afficher toutes les informations nécessaires lorsqu’on vérifie une séance
  4. Le code de service est de 7 chiffres
  5. Afficher la raison pour laquelle un membre serait non validé pour l’accès

Hypothèses

J'ai supposé que le centre sportif #gym possède un système central (le centre de données #GYM) qui enregistre des nouveaux membres et des professionnels par l'intermédiaire d'un agent. Ce centre de données permet au membre de consulter et s'inscrire et comfirmer sa présence à un service offert. Il permet aussi à un professionnel de créer une séance et consulter les inscriptions. De plus, il tient compte des paiements par les membres et les paiements hebdomadaires aux professionnels. Il rédige aussi un rapport destiné au gérant du #GYM.

Tâche 1 : Analyse

Le système qui satisfait les exigences est de mode interactif. Le système requiert de l’assistance manuelle pour accomplir ses tâches et de ressources humaines comme l’agent. Le système est traité par un ordinateur. Il faut donc une base de données pour créer le Centre de Données #GYM et le Répertoire des Services. Le système a donc besoin de disques dur interne et externe suffisamment grand pour stocker les informations d’objets tels que les comptes des membres, les sessions, leurs inscriptions et confirmations, les fichiers TEF, etc. Il faut aussi une imprimante pour imprimer les rapports hebdomadaires pour le gérant. Le système interagit aussi avec les banques pour envoyer les paies.
